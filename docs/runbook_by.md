## Смоук-тэст у тэстнэце Nile

Мэта: даказаць, што ўвесь паток працуе скразным чынам да выхаду ў мэйннэт.

### Папярэднія патрабаванні

- [ ] Паднятая БД тэстнэта Nile (Neon або лакальны postgres)
- [ ] TRON-акаўнты тэстнэта Nile:
  - Згенераваны дэпазітны xprv (тэставыя манеты не маюць рэальнай каштоўнасці, але выкарыстоўвайце свежы ключ)
  - Прыватны ключ гарачага кашалька (папоўнены ≥100 Nile TRX + 100 Nile USDT)
  - Адрас халоднага кашалька (толькі адрас, ключ у env не патрэбны)
- [ ] API-ключ TronGrid (бясплатнага тарыфу для Nile дастаткова)
- [ ] Падняты Upstash Redis KV (бясплатнага тарыфу дастаткова)
- [ ] Токен Telegram-бота ад @BotFather (выкарыстоўвайце тэставага бота, не прадакшэн)
- [ ] ngrok або падобны тунэль для дастаўкі webhook
- [ ] `.env.local` запоўнены ўсімі абавязковымі зменнымі, плюс:
  - `DEPLOY_URL=<ngrok https URL>`
  - `CRON_SECRET` і `ADMIN_API_SECRET` (згенеруйце праз `openssl rand -hex 32`)
  - `TELEGRAM_WEBHOOK_SECRET` (згенеруйце праз `openssl rand -hex 16`)

### Крок 1: Перадпалётная праверка

```bash
# Праверыць, што асяроддзе запоўнена
npx tsx scripts/e2e-nile.ts
```

Усе праверкі павінны прайсці. Выправіце ўсё, што адзначана, перш чым працягваць.

### Крок 2: Міграцыі + сід

```bash
npx tsx scripts/migrate.mts
npx tsx scripts/seed.ts
```

Праверце:
```sql
SELECT * FROM subscription_plans;   -- 3 радкі
SELECT * FROM commission_config;    -- 1 радок
SELECT * FROM ops_kill_switch;       -- 1 радок (false/false)
```

### Крок 3: Webhook

```bash
npx tsx scripts/setup-telegram-webhook.ts <ngrok https URL>

# Праверка
curl https://api.telegram.org/bot<TOKEN>/getWebhookInfo
```

Дачакайцеся `"ok":true` з `"url"`, які паказвае на ваш ngrok-URL.

### Крок 4: Запуск сервера распрацоўкі

```bash
npm run dev
```

Пакіньце гэты тэрмінал адкрытым. Усе запыты маршрутызуюцца праз ngrok → localhost:3000.

### Крок 5: Смоук-тэст — паток пакупкі

Ад імя тэставага карыстальніка Telegram (не адміна):

1. **Адкрыйце бота**, адпраўце `/start`
   - [ ] Бот адказвае вітальным паведамленнем і інлайн-клавіятурай

2. **Націсніце «Buy access»** або адпраўце `/buy`
   - [ ] Бот дасылае рахунак з дэпазітным адрасам TRC20

3. **Адпраўце USDT** (дакладную суму рахунку) з кашалька тэстнэта Nile на дэпазітны адрас
   - [ ] Запішыце хэш транзакцыі: `____________________`

4. **Пачакайце** ~1–2 хвіліны, пакуль спрацуе cron `scan-payments`
   - [ ] Запусціце ўручную: `curl -H "Authorization: Bearer $CRON_SECRET" http://localhost:3000/api/cron/scan-payments`
   - [ ] Праверце табліцу `invoices`: статус павінен быць `paid`, `paid_tx_hash` запоўнены

5. **Праверце падпіску**
   - [ ] У табліцы `subscriptions` ёсць радок са статусам `active`
   - [ ] `/start` карыстальніка цяпер паказвае «Your access is active until …»

6. **Праверце доступ да канала**
   - [ ] Карыстальнік атрымаў спасылку-запрашэнне або быў дададзены ў канал
   - [ ] Калі гэта запрашэнне ў канал, карыстальнік можа паспяхова далучыцца

### Крок 6: Смоук-тэст — рэферальны паток

1. **Атрымайце рэферальны код L1** ад першага карыстальніка (паведамленне `/start` або `ref_code` у БД)

2. **Другі карыстальнік Telegram** адпраўляе `/start <L1_REF_CODE>`
   - [ ] `users.parent_ref_code` усталёўваецца ў код L1

3. **Другі карыстальнік купляе** (паўтарыце крокі 2–6 з Кроку 5)
   - [ ] У карыстальніка L1 з'яўляюцца радкі камісіі ў `commission_ledger` (level=1)
   - [ ] `/start` карыстальніка L1 паказвае рэферальную актыўнасць

4. **Трэці карыстальнік Telegram** адпраўляе `/start <L2_REF_CODE>` (выкарыстайце код другога карыстальніка)
   - [ ] Трэці карыстальнік купляе
   - [ ] Карыстальнік L2 (другі) атрымлівае камісію L1
   - [ ] Карыстальнік L1 (першы) атрымлівае камісію L2 (level=2)

### Крок 7: Смоук-тэст — падаўжэнне

1. **Першы карыстальнік** (з актыўнай падпіскай) адпраўляе `/renew`
   - [ ] Бот прапануе варыянты тарыфаў для падаўжэння
   - [ ] Выберыце тарыф, аплаціце суму рахунку
   - [ ] Новая падпіска пачынаецца з `ends_at` старой падпіскі (стэкінг)
   - [ ] Час не губляецца

2. **Калі падпіска скончылася**, падоўжыце і пераканайцеся, што `starts_at` = зараз (без стэкінгу)

### Крок 8: Смоук-тэст — сканчэнне + напаміны

1. **Часова ўсталюйце `ends_at` падпіскі** на 1 гадзіну наперад:
   ```sql
   UPDATE subscriptions SET ends_at = now() + interval '1 hour' WHERE status = 'active' LIMIT 1;
   ```

2. **Запусціце expire-access:**
   ```bash
   curl -H "Authorization: Bearer $CRON_SECRET" http://localhost:3000/api/cron/expire-access
   ```

3. **Праверце:**
   - [ ] Напамін у асабістыя дасланы (акно T-1h)
   - [ ] Радок `nudges_sent` існуе з window='T-1h'
   - [ ] Няма заўчаснага выдалення (ends_at усё яшчэ > зараз)

4. **Усталюйце ends_at на зараз:**
   ```sql
   UPDATE subscriptions SET ends_at = now() WHERE id = '<SUB_ID>';
   ```

5. **Зноў запусціце expire-access. Праверце:**
   - [ ] Выканана мяккае выдаленне (бан на 35с + разбан)
   - [ ] Статус падпіскі = `expired`
   - [ ] Напамін паўторна не дасланы (ідэмпатэнтнасць)

### Крок 9: Смоук-тэст — выплата

1. **Карыстальнік L1 задае адрас выплаты:**
   - Адпраўце «Set payout address» (кнопка ў меню бота)
   - Адкажыце адрасам TRC20 у Nile

2. **Праверце, што камісіі гатовы да выплаты:**
   ```sql
   SELECT * FROM commission_ledger WHERE beneficiary_id = '<USER_ID>' AND status = 'payable';
   ```
   Калі нейкія маюць статус `accrued` і прайшлі `unlock_at`, cron выплат пераводзіць іх у `payable`.

3. **Запусціце выплату:**
   ```bash
   curl -H "Authorization: Bearer $CRON_SECRET" http://localhost:3000/api/cron/payout-queue
   ```

4. **Праверце:**
   - [ ] Створаны радок `payout_batches` са `status = 'broadcast'`
   - [ ] `tx_hash` усталяваны ў пакеце
   - [ ] Радкі `commission_ledger` пазначаны як `paid`
   - [ ] У аглядальніку Nile USDT паступіў на адрас карыстальніка L1
   - [ ] Запішыце хэш транзакцыі выплаты: `____________________`

### Крок 10: Смоук-тэст — змятанне (sweep)

1. **Праверце, што на дэпазітным адрасе ёсць USDT:**
   Выкарыстайце TronGrid або TronService бота, каб праверыць адрас з Кроку 5.

2. **Запусціце sweep:**
   ```bash
   curl -H "Authorization: Bearer $CRON_SECRET" http://localhost:3000/api/cron/sweep
   ```

3. **Праверце:**
   - [ ] `invoices.swept` = `true` для аплачанага рахунку
   - [ ] `invoices.sweep_tx_hash` усталяваны
   - [ ] USDT паступіў на халодны кашалёк у аглядальніку Nile
   - [ ] Запішыце хэш транзакцыі змятання: `____________________`

### Крок 11: Адмін + бяспека

1. **Адмін-статыстыка:**
   - [ ] Адмін адпраўляе `/admin` → паказана статыстыка (аплачаныя рахункі, баланс гарачага кашалька, стан kill switch)
   - [ ] Не-адмін адпраўляе `/admin` → няма адказу або «not authorized»

2. **Kill switch — пакупкі:**
   ```bash
   curl -X POST -H "Authorization: Bearer $ADMIN_API_SECRET" \
     -H "Content-Type: application/json" \
     -d '{"buyDisabled":true,"reason":"smoke test"}' \
     http://localhost:3000/api/admin/kill-switch
   ```
   - [ ] Карыстальнік адпраўляе `/buy` → бот кажа, што пакупкі адключаны
   - [ ] Уключыце назад: той жа curl з `"buyDisabled":false`

3. **Kill switch — выплаты:**
   - [ ] Адключыце выплаты, запусціце cron выплат, праверце EMPTY_RESULT
   - [ ] Уключыце выплаты назад

4. **Эндпойнт здароўя:**
   ```bash
   curl http://localhost:3000/api/health
   ```
   - [ ] Вяртае 200 з колькасцю рахункаў у чаканні і станам kill switch

## Чэк-ліст прадакшэн-выкату

### Налада кашалькоў

- [ ] **Халодны кашалёк** створаны афлайн (ізаляваная ад сеткі машына або апаратны кашалёк). Адрас запісаны. Прыватны ключ НІКОЛІ не трапляе на машыну з доступам у інтэрнэт.
- [ ] Згенераваны прыватны ключ **гарачага кашалька**. Папоўнены:
  - [ ] ≥ 2 дзён чаканага аб'ёму камісій у USDT
  - [ ] ≥ 100 TRX на папаўненні газу для змятання
- [ ] **Дэпазітны xprv** згенераваны афлайн. У `TRON_DEPOSIT_XPRV` захоўваецца толькі зашыфраваная форма.
- [ ] `TRON_COLD_WALLET_ADDRESS` адпавядае афлайн халоднаму кашальку.
- [ ] `TRON_HOT_WALLET_PK` адпавядае гарачаму кашальку.

### Налада канала

- [ ] Бот прызначаны адмінам у мэтавым прыватным канале
- [ ] У бота ёсць права `invite users` (для выдачы падпісак)
- [ ] У бота ёсць права `ban users` (для мяккіх выдаленняў пры сканчэнні)
- [ ] Зменная асяроддзя `DEFAULT_CHANNEL_ID` адпавядае каналу

### Telegram-бот

- [ ] Токен бота ад @BotFather (прадакшэн-бот, не тэставы)
- [ ] Webhook зарэгістраваны на прадакшэн-`DEPLOY_URL`
- [ ] `TELEGRAM_WEBHOOK_SECRET` зададзены (выпадковы ≥16 сімвалаў)
- [ ] Каманды бота настроены ў @BotFather:
  - `start - Start the bot`
  - `buy - Buy channel access`
  - `renew - Renew your subscription`
  - `withdraw - Withdraw commissions`

### База дадзеных

- [ ] Паднятая прадакшэн-БД Neon
- [ ] `DATABASE_URL` зададзены (радок падлучэння з SSL)
- [ ] Прыменены міграцыі: `npx tsx scripts/migrate.mts`
- [ ] Прыменены сід: `npx tsx scripts/seed.ts`
- [ ] Правераны `subscription_plans` — цэны і працягласці карэктныя
- [ ] Правераны `commission_config` — парогі тарыфаў, стаўкі, мінімальная выплата карэктныя
- [ ] `ops_kill_switch` = `(false, false)` — няма выпадковых блакіровак

### Разгортванне на Vercel

- [ ] Усе зменныя асяроддзя зададзены ў панэлі Vercel (уключаючы `CRON_SECRET`, `ADMIN_API_SECRET`)
- [ ] Cron-задачы Vercel уключаны і паказваюць на прадакшэн-URL:
  - `scan-payments`: `POST /api/cron/scan-payments` кожную 1 хвіліну
  - `expire-access`: `POST /api/cron/expire-access` кожную 1 хвіліну
  - `payout-queue`: `POST /api/cron/payout-queue` кожныя 5 хвілін
  - `sweep`: `POST /api/cron/sweep` кожныя 10 хвілін
- [ ] У cron-задач настроены загаловак `Authorization: Bearer <CRON_SECRET>`
- [ ] Зборка праходзіць на прадакшэне
- [ ] Няма незакамічаных лакальных змен

### Назіральнасць

- [ ] Настроены Sentry DSN (`SENTRY_DSN` або `next.config.ts` з канфігам `sentry`)
- [ ] Sentry прымае падзеі (выклічце тэставую памылку)
- [ ] Настроены Logtail або эквівалентны сток логаў
- [ ] `/api/health` вяртае 200 з карэктнымі дадзенымі
- [ ] Знешні маніторынг аптайму (Better Uptime / Checkly) апытвае `/api/health`
- [ ] Настроена апавяшчэнне пры адказе `/api/health` не 200

### Лімiты бяспекі

- [ ] `MAX_PAYOUT_PER_TX_USDT` зададзены разумным мяжой (патранзакцыйны засцерагальнік)
- [ ] `MAX_PAYOUTS_PER_HOUR` зададзены разумным лімітам частаты
- [ ] `ADMIN_TG_IDS` зададзены (лікавыя ID карыстальнікаў Telegram праз коску)
- [ ] `ADMIN_API_SECRET` зададзены (выпадковы ≥16 сімвалаў, адрозны ад `CRON_SECRET`)
- [ ] Адмін можа атрымаць доступ да `/admin` і бачыць статыстыку

### Перад запускам

- [ ] Смоук-тэст у тэстнэце Nile завершаны (усе крокі з раздзела вышэй пройдзены)
- [ ] Прадакшэн гарачы кашалёк папоўнены
- [ ] Купіце падпіску самі як першы рэальны карыстальнік — праверце ўвесь паток
- [ ] Сябар/калега купляе як другі карыстальнік па вашым рэферальным кодзе — праверце камісію
- [ ] Маніторце логі ў першыя 24 гадзіны на прадмет анамалій
- [ ] Настройце штодзённую зверку камісій (супастаўляйце сумы `commission_ledger.paid` з адтокамі гарачага кашалька)
